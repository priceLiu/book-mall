# 财务 2.0 — 测试用例

> 围绕**视频生成 / 个人 / 团队**，覆盖积分换算、冻结扣费、退款、视频池隔离、团队分账、对账明细与差异、毛利护栏。
> 全部用**库中阿里挂牌价** happyhorse-r2v（净 0.81 元/秒、15s）核算。预期数字见 `财务2.0-验收标准.md` §1。
>
> 落地：纯函数层用 `vitest`（`pnpm --filter book-mall test`，无需 DB）；集成层为带 DB 的脚本/手测步骤。

## A. 纯函数单测（vitest，无 DB）

文件：`book-mall/lib/pricing/__tests__/credit-pricing-formulas.test.ts`、`book-mall/lib/billing/__tests__/reconciliation-diff.test.ts`

### TC-A1 逐档积分换算（核心）
输入 `listPriceYuan=48.6`，对每档单价断言 `computeTierCredits`：

| 单价 | 期望积分 |
|------|----------|
| 0.069000 | 704 |
| 0.049667 | 979 |
| 0.046000 | 1057 |
| 0.042786 | 1136 |
| 0.039967 | 1216 |
| 0.057800（团队高级） | 841 |
| 0.058500（团队豪华） | 831 |
| 0.053267（团队至尊） | 912 |

### TC-A2 逐档毛利恒为 75%
对每档 `computeEffectiveMargin({netCostYuan:12.15, creditsPerUnit:<上表>, pricePerCreditYuan:<单价>})` ∈ [0.749, 0.751]。

### TC-A3 视频 15s 封顶
`videoBillableSeconds(20) === 15`；`videoBillableSeconds(8) === 8`（<15 据实，本期固定 15s 业务侧恒传 15）。

### TC-A4 分类护栏
- `marginGuardFor("PER_SEC") === 0.75`；
- `assertMarginOrThrow(margin=0.667, guard=0.75)` 抛 `MarginGuardError`；
- `assertMarginOrThrow(margin=0.75, guard=0.75)` 通过。

### TC-A5 对账差异纯函数
`diffReconciliation(internal, vendor, tolerance=0.05)`：
- internal=`{happyhorse:121.5}`、vendor=`{happyhorse:121.5}` → `status=OK`、`diffYuan=0`。
- vendor=`{happyhorse:130.0}` → `diffYuan=8.5`、`status=OVER`。
- vendor 缺该项 → `status=MISSING_VENDOR`。

## B. 集成场景（带 DB）

前置：`pnpm --filter book-mall db:deploy && db:seed`，确保 happyhorse-r2v 成本档与个人/团队套餐已落库；视频模型 M=4 已发布。

### TC-B1 个人视频生成 + 冻结扣减（高级版）
1. 用户 U 高级版，视频池余额 6500。
2. 发起 1 条 happyhorse 15s → `reserveCredits(1057)` → 视频池冻结，余额冻结后可用 = 6500−1057=**5443**。
3. 渲染 SUCCEEDED → settle → 写 `creditsCharged=1057`、`costSnapshotYuan=12.15`、`marginSnapshot≈0.75`。
4. 断言：视频池余额 5443；`CreditLedger` 有 RESERVE(−1057) 与 SETTLE 对应流水；通用池不变。

### TC-B2 个人退款语义
- 厂商全失败 → `releaseReserved` → 视频池余额回 **6500**，`creditsCharged=0`。
- 用户取消/部分输出 → 不退 → 余额 **5443**。

### TC-B3 视频池隔离
- 通用池充足、视频池=0 → 发起视频被拦截（`InsufficientCredits`）；文本生成仍成功（扣通用池）。

### TC-B4 团队分账（2 席高级版）
1. 团队 T 共享视频池 = 5000×2=10000。
2. 成员 A 生成 2 条、成员 B 生成 1 条（每条 841）→ 共享池扣 2523。
3. `buildTeamCreditBill({tenantId, periodKey})` 断言：
   - 总消耗 2523；
   - 成员 A 消耗 1682 / 2 次、成员 B 消耗 841 / 1 次；
   - byModel 归口到 `happyhorse-r2v`。

### TC-B5 对账明细与差异
1. 当月团队生成 3 条 → 内部成本 Σ = 12.15×3 = **36.45 元**。
2. `reconcileVendorBill({periodKey, vendorRows:[{vendorModelName:"happyhorse-r2v", amountYuan:36.45}]})` → `status=OK`、`diffYuan≈0`。
3. 改 vendorRows 金额=39.00 → `diffYuan=+2.55`、`status=OVER`。
4. `reconcileByChannel` 按渠道（CHANNEL）核对内部 channelSnapshot 聚合。

### TC-B6 视频专项限流
- 连续发起 3 条并发 → 第 3 条被并发上限（2）拦截。
- 单次批量提交 6 集 → 被批量上限（5）拦截。
- 5 分钟内提交 ≥10 条 → 触发 15 分钟冷却，新任务被拒。

### TC-B7 调价测算护栏
- 把视频 M 由 4 调到 3（毛利 66.7%）→ 发布/测算报告标记不通过、保存被锁。

## C. 回归与兜底
- `CREDIT_BILLING_OFF=1` → 全部积分逻辑短路，不影响生成。
- 幂等：同一 `gateway_log:<id>` 重复结算不重复扣；`reserve:<id>`/`refund:<id>` 重复不重复操作。

### TC-D1 scenario-lab 个人 30 模型
- 运行 `test/unit/scenario-lab.test.ts`：个人高级版 30 行毛利 ∈ **[74.8%, 75.1%]**（护栏 75% 减 **0.2% 取整容差**）。

### TC-D2 scenario-lab 团队 4 席
- 同上：团队高级版 4 席 30 行毛利 ∈ **[74.8%, 75.1%]**。
- 集成：`test/integration/scenario-lab.integration.ts` — 个人 + 团队各 mock settle **30 次**（合计 60 次），逐行毛利校验。
