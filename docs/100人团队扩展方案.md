# 画布视频生成 · 团队/个人并发扩展方案

> 适用：15～100+ 人同时生视频（团队共用 Tenant 或大量个人账号并行）  
> 代码基线：2026-06-19（poll 并行/分片、`TEAM_MAX_CONCURRENCY_CAP`、火山视频 45min 超时）

---

## 1. 三层瓶颈模型

| 层级 | 控制点 | 团队 | 个人散客 |
|------|--------|------|----------|
| **提交** | Gateway `Tenant.maxConcurrency` + Redis 租户槽 | 全团队共享 | 每人 tenant 默认 2 |
| **视频风控** | `VIDEO_*` + 团队对齐 `maxConcurrency` | 共享计费账户 | 每人 2 并发 / 10 排队 |
| **状态同步** | poll worker（DB 扫 SUBMITTED → 调 Gateway） | 全站共享 | 全站共享 |
| **厂商** | 火山 BYOK 配额 | 共用 Key 易撞限 | 建议每人一把 Key |

**结论**：代码架构支持水平扩展；100+ 时 poll worker 与火山配额是主要工程关注点，不是 canvas-web 单线程。

---

## 2. 架构示意

```
用户（团队 N 席 / 个人 × M）
    ↓ SSO + Gateway sk-gw
book-mall：提交任务 → SUBMITTED（PostgreSQL）
    ↓
Poll Worker（并行 + 多轮 + 可选分片）
    ↓ recordInfo
火山 / KIE / 百炼（BYOK）
    ↓ 成功
OSS 持久化 + 回写画布 JSON
```

## 3. 环境变量一览

### 3.1 必开（任何 ≥15 人规模）

```bash
REDIS_URL=redis://...
CANVAS_VOLCENGINE_VIDEO_TIMEOUT_MIN=45
```

未配 `REDIS_URL` 时租户/视频并发限流**全部放行**，峰值易打满厂商。

### 3.2 Poll Worker 扩缩容

| 变量 | 默认 | 说明 |
|------|------|------|
| `GENERATION_POLL_BATCH` | 100 | 每轮 DB 取出 SUBMITTED 上限 |
| `GENERATION_POLL_CONCURRENCY` | 25 | 单 tick 并行 poll 路数 |
| `GENERATION_POLL_MAX_PASSES` | 10 | 单次 cron 最多连续扫描轮数 |
| `GENERATION_POLL_TIME_BUDGET_MS` | 50000 | 单次 HTTP 时间预算 |
| `GENERATION_POLL_MAX_DURATION_SEC` | 300 | poll API `maxDuration` |
| `GENERATION_POLL_SHARD_COUNT` | 1 | 分片总数 |
| `GENERATION_POLL_SHARD_INDEX` | 0 | 当前实例分片号（0…N-1） |

理论单次 cron 扫描量 ≈ `BATCH × PASSES`（受 50s 预算限制）。

### 3.3 团队并发

| 变量 | 说明 |
|------|------|
| `TEAM_MAX_CONCURRENCY_CAP` | 突破套餐档 35 封顶（如 150） |
| `VIDEO_TEAM_QUEUE_MULTIPLIER` | 团队排队 = maxConcurrency × 倍数（默认 5） |
| `CANVAS_PROJECT_INFLIGHT_MAX` | 单画布进行中任务上限（建议 5） |

新建团队默认：`maxConcurrency = min(席位数, max(档位上限, TEAM_MAX_CONCURRENCY_CAP))`。

存量团队同步：

```bash
pnpm --dir book-mall exec tsx scripts/sync-team-max-concurrency.ts
```

## 4. 按规模部署建议

| 同时 SUBMITTED（约） | Poll 部署 | 团队 env |
|---------------------|-----------|----------|
| **≤20** | SCF 30s **或** poll-loop 5s | 默认 + Redis |
| **50** | poll-loop 5s + 2 分片 SCF | `TEAM_MAX_CONCURRENCY_CAP=60` |
| **100** | poll-loop 5s + **4 分片** | `CAP=150`，`CONCURRENCY=30` |
| **200+** | 8 分片 + 调大 BATCH/PASSES；评估消息队列 | 企业档位 + 多火山 Key |

### 4.1 Poll-loop 侧车（推荐 ≥20 人）

在 CloudBase 另起一服务，仅运行：

```bash
CANVAS_POLL_INTERVAL_MS=5000 pnpm --dir book-mall canvas:poll-loop
```

与主 book-mall API 进程解耦，不占用用户请求配额。

### 4.2 SCF 水平分片

复制 `book-mall-canvas-kie-poll` 为 N 个函数，共享 book-mall env：

```bash
GENERATION_POLL_SHARD_COUNT=4
# 各函数分别设
GENERATION_POLL_SHARD_INDEX=0   # 1, 2, 3
```

Cron：每 30 秒。详见 `deploy/tencent/scf/README.md`。

## 5. 现阶段推荐：20 人最优方案

> **场景**：20 人团队同时各生 1 条 Seedance 15s 视频，或约 20 个个人同时用。

### 5.1 目标

- 20 路 Gateway 并发同时提交
- poll 在 3～5s 内扫到全部 SUBMITTED
- 单条最长等待 45min，超时带诊断

### 5.2 book-mall 生产 env（可直接粘贴）

```bash
# 基础
REDIS_URL=redis://<你的 Redis>
CANVAS_VOLCENGINE_VIDEO_TIMEOUT_MIN=45

# 20 席团队：Gateway + 视频风控并发
TEAM_MAX_CONCURRENCY_CAP=25
# 团队开通/续费后跑一次 sync（或后台把 maxConcurrency 设为 20）

# Poll（默认已够用，显式写便于运维）
GENERATION_POLL_BATCH=100
GENERATION_POLL_CONCURRENCY=20
GENERATION_POLL_MAX_PASSES=5
GENERATION_POLL_TIME_BUDGET_MS=50000

# 单画布防单人占满
CANVAS_PROJECT_INFLIGHT_MAX=5

# 个人散客为主时可不设 TEAM_MAX_CONCURRENCY_CAP
# VIDEO_MAX_CONCURRENCY=2
# VIDEO_MAX_QUEUE=10
```

### 5.3 部署

| 项 | 20 人推荐 |
|----|-----------|
| Poll | **poll-loop 侧车** `CANVAS_POLL_INTERVAL_MS=5000`（优先于仅 SCF 1min） |
| SCF | 保留 canvas/story poll 作兜底，Cron **30s** |
| 分片 | **不需要**（`SHARD_COUNT=1`） |
| 火山 Key | 团队：每渠道/每成员 BYOK；避免 20 人共 1 Key |
| DB | 部署迁移 `20260619120000_generation_poll_index` |

### 5.4 验收清单

- [ ] 20 人同时点生成，Gateway RUNNING ≤20，无大量 429
- [ ] `CanvasGenerationTask` SUBMITTED 行 `lastPolledAt` 5s 内更新
- [ ] 45min 内完成则 SUCCEEDED；超时 `failCode` 为 `timeout_vendor_running` 等诊断码
- [ ] Redis 已配，`isConcurrencyEnabled()` 为 true

### 5.5 20 人 vs 100 人差异（备忘）

| | 20 人 | 100 人 |
|---|-------|--------|
| `TEAM_MAX_CONCURRENCY_CAP` | 25 | 150 |
| Poll | poll-loop 5s | poll-loop + **4 分片 SCF** |
| `GENERATION_POLL_CONCURRENCY` | 20 | 30～40 |
| 火山 | 1～2 把 Key 可能够 | 必须多 Key / 提厂商配额 |
