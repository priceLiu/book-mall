# 画布视频生成 · 并发能力说明

> **原则**：人数不是固定值（20、50、100 都可能）。**代码里写好默认值**，生产尽量 **少配 env**。  
> 目标：≥20 人同时生视频不出问题；人数再多随规模自然扩展。

---

## 1. 生产你要做什么（极简）

| 步骤 | 说明 |
|------|------|
| **1. 部署 book-mall** | 含 poll API、45min 火山视频超时、并行 poll worker（已内置） |
| **2. 保持 poll 在跑** | 二选一即可：现有 SCF 调 `/api/canvas/kie/poll`（建议 **30s**）；或侧车 `pnpm canvas:poll-loop`（本地 dev:all 已带） |
| **3. 存量大团队（可选）** | 若团队 `maxConcurrency` 仍是旧默认 2：`pnpm --dir book-mall exec tsx scripts/sync-team-max-concurrency.ts` |

**不必**为一二十人单独调 `GENERATION_POLL_*`、`TEAM_MAX_*`、`VIDEO_*` 等 — 下面表里的值已在代码默认生效。

---

## 2. 代码内置默认（不用配 env）

| 能力 | 默认行为 |
|------|----------|
| **团队 Gateway 并发** | `Tenant.maxConcurrency` = **席位数**（≥20 席保底 20）；**当前未在 Gateway 入口强制**，仅作配置与后续 DB 限流参考 |
| **团队视频风控** | 与 `Tenant.maxConcurrency` 一致的限额 **已定义**；并发/排队 **尚未接入**（见下文「控流方案」） |
| **个人账号** | 每人 2 路视频并发、10 排队（20 个个人 ≈ 40 路） |
| **Poll worker** | 每轮 100 条、并行 25、单次最多扫 10 轮、超时预算 50s |
| **火山 Seedance 超时** | 45 分钟 |
| **单画布进行中任务** | 每项目最多 5 条（防一人占满） |
| **用户总进行中任务** | 50 |

人数从 20 增到 50、100：**团队加席即加并发**；个人场景靠「每人 2 路 × 人数」；poll 默认 batch 100 可继续扛，极端高峰见附录。

---

## 3. 三种用法怎么理解

| 场景 | 系统怎么撑 |
|------|------------|
| **20 人团队** | 20 席 → `maxConcurrency=20`；poll 默认够用 |
| **20 个个人朋友** | 每人 tenant 2 路（限额定义）；实际靠 poll + 火山 BYOK |
| **50 / 100+ 人** | 团队：席位数同步放大；poll 仍用默认，仍不够再开分片（附录） |

硬顶通常在 **火山 BYOK 配额**（多人共用一把 Key）和 **Poll/DB 吞吐**，不在画布前端。

---

## 4. 架构（一眼）

```
用户 → Gateway → 任务 SUBMITTED
         ↓
    Poll Worker（并行，默认已调优）
         ↓
    火山 / OSS → 画布节点更新
```

---

## 5. 验收（20 人规模）

- [ ] 20 人同时点生成，SUBMITTED 任务 `lastPolledAt` 在数秒内更新
- [ ] 长视频 45min 内完成或带超时诊断码失败

## 附录 A · 高级调优（100+ 或运维定制时才看）

| 变量 | 何时才需要 |
|------|------------|
| `GENERATION_POLL_SHARD_COUNT` / `_INDEX` | 单次 poll 仍吃满 CPU、SUBMITTED 积压 |
| `GENERATION_POLL_BATCH` / `_CONCURRENCY` | 压测后微调吞吐 |
| `TEAM_MAX_CONCURRENCY_CAP` | 要覆盖代码默认的席位逻辑 |
| `CANVAS_PROJECT_INFLIGHT_MAX=0` | 要取消单画布 5 条上限 |
| `CANVAS_VOLCENGINE_VIDEO_TIMEOUT_MIN` | 要改 45min 以外的超时 |
