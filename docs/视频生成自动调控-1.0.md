## 0. 背景与目标

生视频是长耗时（厂商侧常 30s～数分钟）、强并发争用、易卡死的操作。1.0 把最近几天到今晨落地的四块改造收敛为一个统一方案：

1. **动静分离**——把「实时在飞/热数据」与「历史/冷数据」在数据库、API、UI 三层拆开，避免高频读拖垮全表。
2. **交通控（交通式控流）**——用「车间距 + 令牌桶 + 信号灯」三层限流，把用户的并发请求平滑地放给厂商。
3. **6 次自动释放（pre-submit 自愈）**——任务在「出队后、提交厂商前」卡住时，自动释放占用并重排队，最多 6 轮，超出才判失败。
4. **生视频解阻塞**——`SUBMITTED` 即释放交通槽 + 立即推进 backlog，杜绝「队首在等、下游已在厂商 RUNNING」的死锁。

**核心理念**：用户点击「立即响应」（先入队、给可见反馈），厂商侧「平滑可控」（限流防风控/超额），系统侧「自愈优先」（卡住自动重排，而非堆积或假死）。

## 1. 任务状态机（权威）

视频任务（canvas `video-engine` / `ai-video-engine`、story `FRAME_VIDEO`、sbv1 视频）统一走下列状态机：

```
QUEUED ──dispatch(占槽)──▶ DISPATCHING ──createTask 成功──▶ SUBMITTED ──poll──▶ SUCCEEDED | FAILED | CANCELLED
  ▲                              │
  │   瞬时失败/503/DB 繁忙        │ 30s 无进展且未提交成功
  └──────────────────────────────┘  （pre-submit 自愈重排，最多 6 轮）
```

| 状态 | 含义 | Gateway 日志行 | 是否占交通槽（`runningVideoCount`） | 用户可见 |
|------|------|----------------|-------------------------------------|----------|
| `QUEUED` | 已入队，等待出队 | 无（前端显示「排队中」合成行） | 否 | 排队中 |
| `DISPATCHING` | 出队中，正在向厂商 createTask | 可能尚无 | **是** | 准备中 / 派发中 |
| `SUBMITTED` | 已提交厂商，等待出片 | `RUNNING` | **否（提交后立即释槽）** | 生成中 |
| `SUCCEEDED` / `FAILED` / `CANCELLED` | 终态 | 终态 | 否 | 完成 / 失败 / 已取消 |

**关键不变量**：
- 交通槽只统计 **DISPATCHING**（见 `reconcileRunningSlotCounts`）。厂商侧长时间 `RUNNING` 的任务**不再占槽**——这是 1.0 解阻塞的核心。
- 瞬时系统错误（503 / DB 繁忙，`isTransientSystemBusyError`）→ 退回 `QUEUED`，**不**判 `FAILED`。
- 非视频任务或 `TRAFFIC_CONTROL_OFF=1` 时，走旧的 `PENDING` 同步路径，不进入交通控。

## 2. 动静分离（Static-Dynamic Separation）

把「动（热）」与「静（冷）」在三层拆分，让实时刷新只碰小而新的数据集。

| 层 | 动（热） | 静（冷） |
|----|----------|----------|
| Gateway 主表 `GatewayRequestLog` | 在飞（`PENDING`/`RUNNING`）+ 近 **1h** 终态 | 更老终态 → 归档表 `GatewayRequestLogArchive`（cron `hotcold:archive`） |
| Gateway 控制台 UI | `mode=live`（实时 Tab） | `mode=history` + 日期窗（默认近 **31 天**） |
| 画布任务表 `CanvasGenerationTask` | 在飞 + 终态 `updatedAt` 近 **6h** | 更老终态不参与高频 `/tasks` 轻量读 |
| 状态投影 `GatewayStatsCounter` | 增量 bump + **4s TTL** 自愈缓存 | 过期后全量重算 |

**实现要点**：
- 后端 `buildGatewayLogHotWhere()` = `在飞 OR completedAt ≥ cutoff`；`applyGatewayLogQueryMode()` 默认 live 才附加热区 WHERE，history 不附加。
- 前端 live 列表每批 `LOGS_BATCH_SIZE = 30` + 无限滚动（上限 `INFINITE_SCROLL_MAX_ROWS = 500`），并用 `filterLiveHotWindowRows` 剔除过期终态缓存行。
- 「无在飞就不 poll」：`shouldGatewayLivePoll()` 在无热数据时把 interval 归 0、停止请求。
- facets（筛选项）查询与日志行查询解耦（`facets=only`），互不拖累。

**关键常量（代码为准）**：

| 名称 | 默认 | env |
|------|------|-----|
| `getGatewayLogHotRetentionHours()` | 1 | `GATEWAY_LOG_HOT_RETENTION_HOURS` |
| `GATEWAY_HISTORY_DEFAULT_RANGE_DAYS` | 31 | — |
| `LOGS_BATCH_SIZE` / `INFINITE_SCROLL_MAX_ROWS` | 30 / 500 | — |
| `GATEWAY_LIVE_POLL_ACTIVE_MS` | 8000 | — |
| `GATEWAY_LIVE_HOT_SYNC_MS` | 15000 | — |
| `GATEWAY_STATS_DEFAULT_TTL_MS` | 4000 | — |
| `getCanvasTaskTerminalHotHours()` | 6 | `CANVAS_TASK_TERMINAL_HOT_HOURS` |

> 已知不一致：前端 `GATEWAY_LOG_HOT_RETENTION_MS` 硬编码 1h，不读后端 env；本地若把后端调成 24h，会短暂前后端不一致，直到 API 返回 `hotCutoffMs`。

## 3. 交通控（车间距 + 令牌桶 + 信号灯）

入口：`book-mall/lib/generation/traffic-control/`。用户点击后**先入队**（`admitCanvasVideoTask` → `QUEUED` + `trafficStartedAt` + stagger），再由调度器按三层限流出队。

**三层闸门**（`slot.ts` + `token-bucket.ts`）：
1. **车间距**：距上次 dispatch < `ACTOR_DISPATCH_MIN_MS` 则跳过，写 `dispatchAfter` 延后。
2. **令牌桶**：`TRAFFIC_TOKENS_PER_SEC` 限速 + burst；占槽消耗一个令牌。
3. **信号灯**：`runningVideoCount >= maxConcurrency` 则拒绝；原子 `updateMany` 防越额。

**出队**：poll worker `dispatchQueuedCanvasTasks({ fastPath:false })` 或用户点击触发 `fireCanvasDispatchForProject({ fastPath:true })`；每轮先跑 pre-submit 自愈，再 `findMany QUEUED take DISPATCH_BATCH`，逐个 `dispatchOneCanvasQueuedTask`。

**关键常量（`constants.ts` 为准）**：

| 名称 | 默认 | env |
|------|------|-----|
| `isTrafficControlEnabled()` | 开 | `TRAFFIC_CONTROL_OFF=1` 关闭 |
| `getActorDispatchMinMs()` | 1200 | `ACTOR_DISPATCH_MIN_MS` |
| `getActorDispatchJitterMs()` | 2000 | `ACTOR_DISPATCH_JITTER_MS` |
| `getTrafficTokensPerSec()` | 1 | `TRAFFIC_TOKENS_PER_SEC` |
| `getDispatchBatch()` | 5 | `DISPATCH_BATCH` |
| `getDispatchSubmitTimeoutMs()` | 25000（且 < staleSec−2s） | `DISPATCH_SUBMIT_TIMEOUT_MS` |
| `VIDEO_MAX_CONCURRENCY` / `getVideoPersonalMaxConcurrency()` | **6** | `VIDEO_MAX_CONCURRENCY` |
| `getCanvasProjectInflightMax()` | 5（0=不限） | `CANVAS_PROJECT_INFLIGHT_MAX` |
| `getCanvasUserInflightMax()` | 50 | `CANVAS_AI_USER_INFLIGHT_MAX` |
| `CANVAS_DB_TX_OPTIONS` timeout | 30s | — |

**积分**：排队阶段**不** RESERVE；入队前只读预检（`video-queue-precheck`），dispatch 内 createTask 时才扣减。

> 注意分层：`canvas-web/lib/canvas/run-queue.ts` 是**前端单画布** ≤5 并发的本地队列；后端 `GenerationTrafficState` 是**服务端跨用户/租户**的并发控制。两层独立，勿混。
> 已知文档/代码差异：`docs/全站架构图与配置表.md` 旧值（2500ms / 0.5/s）已被 `constants.ts`（1200ms / 1/s）取代，以代码为准。

## 4. 6 次自动释放（pre-submit 自愈 / 自动重排队）

入口：`recover-stale-dispatching.ts`（canvas）/ `recover-stale-dispatching-story.ts`（story）/ `pre-submit-retry.ts`（计数）。

解决「任务出队后占了槽，却因 createTask 超时/瞬断没真正提交厂商」导致的**槽位泄漏 + 队列假死**。

**Stale 判定**（`isStalePreSubmitTask`，cutoff = now − `DISPATCHING_STALE_SEC`）：
- `updatedAt ≤ cutoff`，或
- 无厂商 taskId 且 `dispatchingAt ≤ cutoff`，或
- 无厂商 taskId 且 `queuedAt/createdAt ≤ cutoff`

**卡住时两层释放**：
1. 释放交通槽（`releaseTrafficSlot`）。
2. 清理提交 claim（`gatewayKieSubmitClaimed` 但无 log 时）。

**重排**：→ 回 `QUEUED` + 重设 `dispatchAfter`，并触发 `fireVideoTrafficDispatchBacklog` 推进。

**计数与上限**：
- `dispatchStaleRetryCount` 存于 `inputPayload`，每轮 +1。
- `getDispatchStaleRetryMax()` = **6**（env `DISPATCH_STALE_RETRY_MAX`）。
- 达到 6 轮仍未 `SUBMITTED` → `failCanvasTaskPreSubmitTimeout`，`failCode = SUBMIT_DISPATCH_TIMEOUT`，文案「提交生成超时, 请重试」。
- 出队阶段最长约 30s × 6 ≈ **~3 分钟**。
- 仅 `SUBMITTED` 成功时 `clearDispatchStaleRetryInPayload` 清零；**自愈重排不重置** `trafficStartedAt`（系统总耗时连续计）。

**特殊**：已有 `gatewayLogId` 却仍 DISPATCHING → `promoteDispatchingWithGatewayLog()` 直接补 `SUBMITTED`，避免合成行永久卡住。

**触发入口**：`dispatchQueuedCanvasTasks` 每轮开头、`canvas-tasks` GET、`canvas-queue` GET（`scheduleRecoverStaleDispatching`，同 project 5s 防抖）。

| 名称 | 默认 | env |
|------|------|-----|
| `getDispatchingStaleSec()` | 30 | `DISPATCHING_STALE_SEC` |
| `getDispatchStaleRetryMax()` | **6** | `DISPATCH_STALE_RETRY_MAX` |
| `SCHEDULE_RECOVER_MIN_GAP_MS` | 5000 | — |

## 5. 生视频解阻塞（队首不被下游堵死）

**要解决的现象**：逻辑上第一个排队的任务还在等，下面 6 个却已在厂商 RUNNING——队首被「已提交但未释槽」的下游堵住。

**解法链（事实）**：
1. **`SUBMITTED` 立即释槽** + `fireDispatch:true`（`dispatch-canvas.ts` → `releaseGatewayVideoTrafficSlotIfOccupying`）→ 后续 `QUEUED` 立即可出队，不必等厂商终态或 10min promote。
2. **`reconcileRunningSlotCounts`** 只对齐 DISPATCHING 数量，修复历史泄漏的槽。
3. **Pending 合成行**（`canvas-pending-log-row.ts`）：QUEUED/DISPATCHING 无真实 Gateway log 时，Logs 顶部显示「排队中（待提交）」；真实 log 出现后去重。
4. **`canvas-queue` GET**：有排队 → `reconcileRunningSlotCounts` + `fireVideoTrafficDispatchBacklog` + `scheduleRecoverStaleDispatching` +（`rows=1`）`autoRecoverPollStalledVolcengineGatewayLogs({ limit:6 })`。
5. **厂商分阶段计时**：进行中停更 > 2min → `stalled=true`，冻结「生成中」墙钟，提示「核对厂商」。
6. **厂商停更收口**：`VOLCENGINE_VENDOR_STALE_RELEASE_MS = 600s` 自动收口；`VOLCENGINE_QUEUED_STALE_MS = 10min` 判厂商队列卡死。
7. **10min 后台化**：仍 RUNNING 则释槽并继续 poll（不阻塞用户）。

**相关时间阈值**：

| 场景 | 常量 | 默认 |
|------|------|------|
| pre-submit 自愈周期 / 最大轮次 | `DISPATCHING_STALE_SEC` / `DISPATCH_STALE_RETRY_MAX` | 30s / 6 |
| createTask HTTP 超时 | `DISPATCH_SUBMIT_TIMEOUT_MS` | 25s |
| 出队前 UI 告警（久未 poll） | `POLL_INFLIGHT_WARN_MS` | 120s |
| 厂商停更/轮询失联（冻结墙钟） | `VOLCENGINE_*_POLL_GAP_MS` | 2min |
| 自动「核对厂商」节流 | `AUTO_RECOVER_MIN_GAP_MS` | 30s/行 |
| 厂商 running 停更收口 | `VOLCENGINE_VENDOR_STALE_RELEASE_MS` | 600s |
| 视频业务硬超时 | `getCanvasVolcengineVideoTimeoutMin()` / `STALE_TIMEOUT` | 45min / 90min |

> 说明：用户观察到的「系统总耗时 400~600s」并非单一硬阈值——「生成中」墙钟由前端实时递增；600s 对应厂商停更自动收口（`VOLCENGINE_VENDOR_STALE_RELEASE_MS`）。代码中没有「出队前 60~76s」的硬上限，那是 pre-submit 自愈周期叠加的观测值。

## 6. 收口公平化与厂商差异化轮询（poll 不饿死 + 利用火山并发）

> 解决现象：Gateway Logs 出现「厂商早已出片，但系统 50~76min 后才收口」「Gateway 分段 > 总耗时」。

### 6.1 根因：poll worker 尾部饿死

旧 `runGatewayPollWorker` 每 10s tick 取「最旧 `submittedAt` 的前 30 条」并 `take:30`。当在飞 RUNNING > 30（多人同时生视频，尤其火山 ~50 并发）时，**排在 30 名以外的任务永远轮不到 poll**，直到旧任务终态腾位才被看见——于是厂商 5min 出片、系统几十分钟才收口；又因画布侧 45min 硬超时（`getCanvasVolcengineVideoTimeoutMin()`）封顶 `e2eMs`，而 gateway 仍在超时后继续 poll，造成「Gateway 分段 > 总耗时」的观测异常。

### 6.2 公平调度（L1）

poll 选取从「最旧 `submittedAt`」改为「**最久未 poll**」：

```
ORDER BY lastPolledAt ASC NULLS FIRST, submittedAt ASC
```

`NULLS FIRST` 让从未 poll 的新任务最优先；其余按 `lastPolledAt` 升序，保证**每个在飞任务的最大 poll 间隔 ≈ 一个 tick**，杜绝尾部饿死。常量见 `POLL_FAIRNESS_ORDER`（`lib/gateway/poll-service.ts`）。

### 6.3 厂商差异化机制（完成感知方式分流）

不同厂商「如何知道视频已生成」根本不同，poll 预算据此分流：

| 厂商类 | 代表 | 完成感知 | poll 角色 | tick 预算（默认） |
|--------|------|----------|-----------|-------------------|
| **纯轮询 · 高并发** | **VOLCENGINE 火山** | 无回调，必须主动轮询；账号支持 ~50 并发 | **主力**：最大预算 + 公平调度 | `GATEWAY_POLL_VOLCENGINE_BUDGET=60` |
| **回调驱动** | **KIE** | `callBackUrl` 推送（`/api/canvas/kie/callback/[kind]` → `applyCanvasKieTaskResult`）即时收口 | **仅兜底**：只补捞「久未 poll」的少量，省预算给火山 | `GATEWAY_POLL_CALLBACK_BACKSTOP=8`（仅 `lastPolledAt` > 90s 的） |
| **其它异步轮询** | BAILIAN / DASHSCOPE / HUNYUAN | 轮询 | 中等预算 + 公平调度 | `GATEWAY_POLL_OTHER_BUDGET=24` |

要点：
- 火山纯轮询，是收口延迟的「重灾区」，给最大预算；KIE 正常由回调收口，poll 只在**漏回调**时低频补捞，因此 KIE 不再与火山抢 30 条预算。
- 三类一次性取出后合并、火山优先入队，在 `GATEWAY_POLL_CONCURRENCY=8` 有限并发下统一 poll，单条 hung 调用（20s 超时）不拖垮整 tick。

### 6.4 兜底安全网（L2）

worker 每个 tick 末尾内置调用 `autoRecoverPollStalledVolcengineGatewayLogs({ staleMs, limit })`：任何 **2min（`GATEWAY_VOLCENGINE_STALL_RECOVER_MS`）** 未被 poll 到的火山 RUNNING，主动向厂商核对并收口——**不再依赖用户打开 Logs 页**触发。公平调度下正常任务 `lastPolledAt` 是新的，不会被此处重复 poll；它只兜「预算被打满（在飞 > 60）」的极端尾部。

### 6.5 利用火山 ~50 并发

- **poll 侧（已落地）**：火山预算默认 60 ≥ 并发上限 50，配合公平调度 → 50 个在飞每 tick 全部被 poll，最大收口延迟从「几十分钟」降到「≈ 一个 tick（~15~25s）」。这是「用满 50 并发」的前提：没有它，50 并发只会制造 50 个饿死的尾部。
- **dispatch 侧（设计，待定）**：当前交通槽只计 DISPATCHING、`SUBMITTED` 即释槽，故厂商侧 RUNNING **无硬上限**——超过 50 时由 pre-submit 6 次自愈吸收厂商超额报错并重排，不会雪崩，但有少量空转。建议增设**按 BYOK 凭证维度**的在飞上限 `CANVAS_VOLCENGINE_INFLIGHT_CEILING≈45`（留 50 的余量）：dispatch 前统计该凭证 RUNNING 火山视频数，≥ 上限则 `dispatchAfter` 延后。**暂不硬编码**，因「50 是按账号还是按单把 sk-gw key」尚未与厂商确认；按账号则需全局计数，按 key 则需按 `gatewayCredentialId` 计数，语义错会误限流。确认后在 `dispatchOneCanvasQueuedTask` 的 VOLCENGINE 分支前置该闸门。

### 6.6 三道收口（确保「厂商已生成」必被收口）

1. **回调即时收口**（KIE）：厂商出片 → callback → `applyCanvasKieTaskResult`，毫秒级收口；poll 仅兜底。
2. **公平轮询收口**（火山/其它）：每 tick 全量公平 poll，厂商终态后 ≈ 一个 tick 内 `SUCCEEDED`/`FAILED` 落库 + OSS 回写。
3. **停更兜底收口**（火山）：L2 2min 安全网 + `VOLCENGINE_VENDOR_STALE_RELEASE_MS=600s` 停更自动收口 + 45/90min 业务硬超时，确保任何遗漏最终收口、不留孤儿 RUNNING。

### 6.7 关键常量（`poll-service.ts` 为准）

| 名称 | 默认 | env |
|------|------|-----|
| 火山 poll 预算 | 60 | `GATEWAY_POLL_VOLCENGINE_BUDGET` |
| 其它轮询厂商预算 | 24 | `GATEWAY_POLL_OTHER_BUDGET` |
| 回调厂商兜底预算 | 8 | `GATEWAY_POLL_CALLBACK_BACKSTOP` |
| 回调兜底「久未 poll」阈值 | 90s | `GATEWAY_POLL_CALLBACK_BACKSTOP_STALE_MS` |
| poll 并发 | 8 | `GATEWAY_POLL_CONCURRENCY` |
| L2 火山兜底 stale 阈值 / 条数 | 120s / 12 | `GATEWAY_VOLCENGINE_STALL_RECOVER_MS` / `GATEWAY_VOLCENGINE_STALL_RECOVER_LIMIT` |
| 火山在飞上限（设计，未启用） | 45 | `CANVAS_VOLCENGINE_INFLIGHT_CEILING` |

### 6.8 两个 poll worker 与生产部署（重要）

收口由**两个独立 worker** 协作，二者**都已 `lastPolledAt` 公平调度**：

| worker | 轮询对象 | 生产驱动 | 本地驱动 | 关键 env |
|--------|----------|----------|----------|----------|
| **`runCanvasPollWorker`**（画布/漫剧任务收口主力） | `CanvasGenerationTask` + 顺带核对厂商 recordInfo、收口 Gateway 日志 | **SCF `POST /api/canvas/kie/poll`**（每 1min，长视频建议 30s） | `pnpm canvas:poll-loop` | `GENERATION_POLL_*` |
| **`runGatewayPollWorker`**（本次 §6 改造） | `GatewayRequestLog`（外部 gateway 任务、story、兜底） | gateway-poll 侧车 / 仪表盘 opportunistic | `gateway-poll-loop`（dev:all） | `GATEWAY_POLL_*` |

- 画布视频在生产**主要由 `runCanvasPollWorker` 收口**，其候选选取已用 `ORDER BY lastPolledAt ASC NULLS FIRST`（`canvas-task-service.ts`），默认 `GENERATION_POLL_BATCH=100` / `GENERATION_POLL_CONCURRENCY=25`。
- **50~100+ 并发的生产关键开关是分片**：把 `book-mall-canvas-kie-poll` SCF 复制 4 份，book-mall env 设 `GENERATION_POLL_SHARD_COUNT=4`，各函数 `GENERATION_POLL_SHARD_INDEX=0..3`，Cron 均每 30s（见 `deploy/tencent/scf/README.md`）。这是「用满火山 ~50 并发又不让收口饿死」在生产的主手段。
- `GATEWAY_POLL_*`（§6.7）作用于 `runGatewayPollWorker`：本地 dev:all 必经；生产若跑 gateway-poll 侧车/SCF 才生效，对外部 gateway 任务与兜底有意义。

## 7. 影视专业版 2.0（Pro2）落地约定

Pro2 的生视频节点（sbv1 视频引擎、分镜视频）**复用同一套后端调控**，禁止自建调度：

1. **入队**：视频生成统一调 `admitCanvasVideoTask`（或 story/sbv1 对应 runner，最终汇入 traffic admit），节点 runtime 先进入 `pending/queued`，UI 立即给可见反馈。
2. **状态映射**：节点 runtime 状态须能映射到 §1 状态机；前端不得把 `QUEUED/DISPATCHING` 当失败处理（参考 `canvas-run-session.ts`、`hydrate-inflight-preserve.ts` 的「在飞优先、不被旧任务回刷」逻辑）。
3. **并发**：受后端 `VIDEO_MAX_CONCURRENCY=6`（个人）与前端单画布 ≤5 共同约束；Pro2 不得绕过 traffic-control 直连厂商（见 `canvas-gateway-no-direct-connect.mdc`）。
4. **自愈**：依赖后端 pre-submit 30s×6 自愈与 `SUBMITTED` 即释槽；Pro2 前端只需轮询 `/canvas-queue`、`/canvas-tasks` 并按返回 reconcile，不要自己写「卡住重发」。
5. **错误**：`SUBMIT_DISPATCH_TIMEOUT`（提交超时）、`INSUFFICIENT_CREDITS`（积分不足）等 failCode 走统一的 `friendly-task-error.ts` + 弹层标题（`libtv-runtime-error-alert.ts`），避免「画布闪」。
6. **日志可观测**：Pro2 的视频任务在 Gateway Logs 走动静分离 live 视图；排队/派发阶段以 pending 合成行呈现，提交后并入真实日志行。
7. **模型选择**：模型与可调参数仍走 `EnginePicker`（见 `pro2-model-picker.mdc`），调控方案与模型选择解耦。
