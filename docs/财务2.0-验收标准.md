## 0. 固化口径（验收前提）

| 项 | 取值 | 来源 |
|----|------|------|
| 积分换算模型 | 方案 B-refined：逐档真实单价派生，单一元计价真值 | 决策固化 |
| 单一真值 | `listPriceYuan = 净成本 × M`（元计价，存 `ModelCreditPrice`） | `lib/pricing/credit-pricing-engine.ts` |
| 逐档积分 | `credits = round(listPriceYuan ÷ 该档单价)`；单价 = `套餐价 ÷ 月积分` | 本期新增 `computeTierCredits` |
| 视频系数 M | 4 → 恰好 75% 毛利 | `定价与风控.md` |
| 文本/图像 M | 默认 2.5 → 60%，后台可配 | 现网 |
| 毛利护栏 | 按计费类型分类：视频 ≥0.75；文本/图像 ≥0.30（可配） | 本期新增 |
| 退款语义 | 视频先冻结后渲染；用户取消/部分输出/不满意不退，仅厂商全失败全额退；文本/图像保留「失败即退」 | 决策固化 |
| 视频时长 | 固定 15s，按秒计费封顶 15s | `定价与风控.md` |
| 核算基准模型 | `happyhorse-r2v`（aliyun, PER_SEC, list 0.9, 折扣 0.1 → **净 0.81 元/秒**） | `lib/billing/seed-credit-billing.ts` |

> 库中阿里视频仅 happyhorse 单档（无 720P/1080P 拆分）。本期按此核算；若上线真值 720P/1080P，另加 `ModelCostProfile` 成本档，验收数字按相同公式重算。

## 2. 各能力 Done-Criteria

### 2.1 计费引擎（Phase 1）
- [ ] `computeTierCredits(listPriceYuan, pricePerCreditYuan)` 输出 §1 表中数字。
- [ ] 视频按 15s 封顶计费：`durationSec>15` 仍按 15 计。
- [ ] 发布报价时，视频模型 M<4（毛利<75%）被 `MarginGuardError` 拦截。

### 2.2 视频双池 + 冻结扣费（Phase 2）
- [ ] `CreditAccount` 含 `videoBalanceCredits` / `videoMonthlyGrant` / `pricePerCreditYuan` 快照。
- [ ] 发起视频：先 `reserveCredits`（冻结，写 RESERVE 流水），余额不足拦截。
- [ ] SUCCEEDED → 冻结转扣减（SETTLE），写 `costSnapshotYuan=12.15`、`marginSnapshot≈0.75`。
- [ ] 厂商全失败 → `releaseReserved` 全额返还，余额复原。
- [ ] 用户取消/部分输出/不满意 → 冻结转扣减，不退。
- [ ] 视频走视频池、文本/图像走通用池，互不串池。

### 2.3 视频专项风控（Phase 3）
- [ ] 单账号 2 并发、队列上限 10。
- [ ] 单日视频条数上限（按档），超限拦截。
- [ ] 单次批量 ≤5 集；5 分钟累计 ≥10 条触发 15 分钟冷却。
- [ ] 异常用户阶梯管控（弹窗→次日减半→关批量→封视频权限）可触发与查询。

### 2.4 调价测算 + 反向验算 + 审批流（Phase 4）
- [x] 改套餐参数触发六维测算报告（基础成本/扣费变动/毛利校验/月度成本上限/营收模拟/用户影响）。
- [x] 任一场景毛利 < 阈值 → 报告标记「校验不通过」，锁定保存。
- [x] 反向验算模式 A（目标毛利反推定价）/ 模式 B（保本线核验）输出整改建议。
- [x] `PlanChangeProposal` 全流程留痕：运营提交→财务复核→超管终审→`applyEffectiveProposal` 生效。

## 3. 对账验收（围绕视频）

- [ ] 内部成本：当月 happyhorse 视频 N 条 → Σ `costSnapshotYuan` = 12.15 × N。
- [ ] 厂商账单 = 12.15×N → 对账 `status=OK`（容差 5%）。
- [ ] 厂商账单偏高（如单条 13.00）→ `diffYuan>0`、`status=OVER`。
