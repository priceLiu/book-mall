# 正式接入支付渠道（以支付宝扫码为例）

> 本文说明从当前 **模拟收银** 过渡到 **真实收款**（如支付宝当面付 / 扫码支付）时，工程与业务上**必须注意**的事项。具体接口字段以支付宝开放平台当期文档为准。

## 1. 上线前置：关闭模拟能力

- **生产环境**不得设置 **`ALLOW_MOCK_PAYMENT=true`**，并确保 **`/api/dev/mock-*`** 在非 development 下不可用（当前实现已依赖环境与该开关）。
- 前端入口应从 **`/pay/mock-*`** 切换为 **渠道收银页 / 真实二维码页**，避免用户误以为仍为演示。
- 模拟落库逻辑（`lib/apply-mock-subscription.ts`、`lib/apply-mock-topup.ts`）仅作过渡参考；真实支付应在 **独立适配层** 中完成后调用「业务入账」函数，避免复制粘贴散落多处。

## 2. 可信来源：异步通知优先于前端提示

扫码支付的典型路径：**服务端下单 → 返回二维码 → 用户支付宝付款 → 支付宝服务端异步通知（notify_url）**。

| 做法 | 说明 |
|------|------|
| **以异步通知为支付成功依据** | 用户关掉页面、网络抖动时，浏览器回调不可靠；**必须在服务端验签后的 notify 里**完成订单状态变更与权益发放。 |
| 同步跳转（return_url） | 仅改善体验（展示「支付成功」），**不得**作为唯一入账依据；可与 notify **幂等配合**（同一笔交易多次处理结果一致）。 |
| **验签** | 对支付宝 POST 正文使用官方算法验签；失败则返回非 success，并按文档重试策略处理。 |

## 3. 幂等与重复通知

支付宝可能对同一笔支付 **多次投递** notify。

- 业务更新（订单置 PAID、写订阅、加余额）必须 **幂等**：以支付宝 **`trade_no`**（或等价全局唯一交易号）+ 本方 **`out_trade_no`**（商户订单号）做唯一约束或「已处理则直接返回 success」。
- 数据库建议：`Order`（或支付流水表）上对 **`channelTradeNo`** / **`outTradeNo`** 唯一索引，或在事务内 **先查再更新**。

## 4. 金额与安全校验

- notify 中的 **金额、币种、卖家账号** 必须与 **本方下单时记录** 一致后再入账；防止篡改请求伪造成功。
- **`out_trade_no`** 必须在服务端生成并与登录用户、业务类型（订阅 / 充值）、金额绑定；**禁止**由客户端随意传入未校验的金额。
- **密钥**：应用私钥、支付宝公钥仅保存在服务端环境变量或密钥管理服务；**绝不**进入前端 bundle 或仓库。

## 5. 订阅 vs 钱包充值（业务分支）

与本站现有模型对齐时建议区分：

| 类型 | 要点 |
|------|------|
| **订阅** | 支付成功后：激活/续期 `Subscription`，写 `Order(type: SUBSCRIPTION)`；周期起止与套餐一致；注意 **续费** 与 **新订**、并发支付同一用户的策略。 |
| **钱包充值** | 支付成功后：`Wallet` 增加余额、`WalletEntry(RECHARGE)`、`Order(type: WALLET_TOPUP)`；与产品规则「订阅与余额互不抵扣」一致（见 `doc/product/02-users-billing-and-balance.md` 等）。 |

同一套 notify 入口可按 **`out_trade_no` 前缀或订单类型字段** 分发到不同入账逻辑。

## 6. 二维码与订单生命周期

- **下单未支付**：订单状态可为 `PENDING`（若引入）；超时 **关单** 策略需与支付宝侧一致，避免用户扫过期码。
- **二维码展示**：URL / code 由官方接口返回；注意 **有效期** 与前端倒计时提示。
- **用户取消 / 支付失败**：不改变业务权益；仅在前台提示，后台保留审计流水。

## 7. 环境与密钥隔离

- **沙箱**与**生产**使用不同 AppId、密钥与网关地址；部署配置（Vercel Environment）按 **Preview / Production** 分离。
- 轮换密钥时规划 **双密钥并行窗口**，避免全场支付中断。

## 8. 部署形态（如 Vercel）

- **notify URL** 必须为公网 HTTPS，且不会被 CDN 错误缓存 POST。
- Serverless 函数 **执行超时** 须大于 notify 处理耗时（验签 + DB 事务）；过重逻辑可写入队列异步处理，但须仍 **快速返回** 支付宝要求的应答（避免重复投递风暴）。
- 若 notify 依赖 DB，连接池与 Neon 冷启动问题仍存在，参见 [`stack-and-environment.md`](../tech/stack-and-environment.md)。

## 9. 对账、提现与客服

- 定期使用支付宝 **对账单** 与本站 `Order` / 流水核对，发现短款或重复入账可追溯。
- **余额提现**：用户对前台的「提现」在渠道侧常对应原路退回等能力；对接时按支付宝等开放接口实现，并与本站 `Wallet` / 订阅周期规则联动（是否允许退订、余额提现路径见产品文档第七章及提现模块）。
- 保留 **商户订单号 ↔ 支付宝流水** 映射，便于客服查询。

## 10. 合规与产品文案

- 支付页与协议中明确：**收款主体**、服务内容、发票（若适用）、提现规则（与前台计费公示一致）。
- 个人信息与支付敏感数据遵循最小化收集；不落库完整支付密码；PCI 范围内责任以支付宝托管为准时亦需在协议中说明。

## 11. 与当前代码的衔接建议

1. 新增 **`lib/payments/`**（或等价目录）：`createPayment`、`handleAlipayNotify`、`fulfillOrder`（内部再调用订阅入账 / 充值入账）。
2. 将现有 **`applyMock*`** 中的 **事务块** 提炼为 **`fulfillSubscriptionPaid`** / **`fulfillWalletTopupPaid`**，供 notify 与人工补单（极少）复用。
3. 过渡期内可保留 mock 路由仅用于 development，与真实路由 **路径、权限完全分离**。
