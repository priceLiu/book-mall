## 1. 你的目标（我们按这个对齐）

- **新项目分开做**：像现在的工具站一样，全模态工作室是 **单独的程序、单独部署**，例如单独二级域名 **`m.ai-code8.com`**（名字可改，模式和 **`tool.ai-code8.com`** 一致）。  
- **入口仍挂在「工具」这一边**：用户在工具站侧栏点进工作室，感觉仍是一家人；背后可以是 **跳转到 `m.ai-code8.com`**。  
- **要轻**：少一点你不需要关心的耦合；但不是「各写各的钱包」那种假轻。  
- **要合理、安全、高效**：该分的地方分，**划账只走一条正门**，别绕小路。  
- **财务必须重构**：从 **用户自己的钱**（钱包、扣费、流水）到 **管理后台**（定价、提现、优惠）再到 **核算对账**（算得清、对得上）—— **都要纳入同一块「财务」规划里重做边界**，而不是只改一个页面。

---

## 2. 三个域名，三台「车」（举例）

| 域名（举例） | 是什么 | 主要负责 |
|--------------|--------|----------|
| **`book.ai-code8.com`** | 主站 | 注册登录、会员充值、个人中心、**财务后台配置与审核**、课程与商城等 |
| **`tool.ai-code8.com`** | 工具站 | 工具列表与壳、**从主站换票登录**、链到各个工具 |
| **`m.ai-code8.com`** | 全模态工作室（新应用） | 工作室里的交互与模型调用；**扣款不落本地库**；**可放充值入口**，到账仍走主站同一套收银/入账 |

以后还有别的新工具，可以继续加 **`xxx.ai-code8.com`**，**套路一样**：独立部署 + 挂到工具站入口。

---

## 3. 「合理的轻」是什么意思（四句话）

1. **谁管钱，谁说了算**：余额、扣款、流水、退款 —— **只在一套财务能力里完成**（先装在主站里整理清楚，将来可以变成独立「财务小服务」，但对外还是这一套能力）。  
2. **工具只管两件事**：① 在用户机器上把事做完；② 需要收钱时，**带齐凭证到主站（财务）说一声「请按规则扣」** —— **不直接改数据库里的余额**。  
3. **少打电话，不是不打电话**：价目可以 **批量拉、短缓存**，不必每一步都问主站；但 **真正动钱那一次**，要 **走正门、可防重复**。  
4. **分开部署，不等于分账各写各的**：`m` 和 `tool` 可以各自扩机器；**账本仍是一座金库**，一个出纳。

## 4. 用户侧财务（钱包视角）

- 用户在 **主站** 充值、看余额、看消费记录、申请提现（个人中心仍以主站为准也可以）。  
- 用户在 **`tool` / `m`** 里用功能时，**钱怎么进、怎么出、怎么判断「够不够」**，下面分三块说清楚。

### 4.1 充值（进账）

- **`tool` / `m` 上可以放「去充值」**：跳转主站收银 / 嵌入收款页等都可以，**目标是少打扰、闭环顺畅**。  
- **账本规则不变**：用户卡里或微信里真收到钱、系统里给用户加余额 —— **仍只走主站（未来统一的财务）这一根管道**。工具站 **不负责**「私自给用户加钱」。  
- 换句话说：**进门可以在边缘应用，`+余额` 的盖章只在一处**。

### 4.2 出账（扣款、与「上报用量」）

- **模型跑出来、要记账扣费时**：无论人在 **`tool` / `m`** 哪一端，**写扣款流水、动可用余额** —— 只通过 **主站财务侧** 的正式接口（今天换票 + 上报用量的语义延续）。  
- 工具进程 **本地不落「真实账本」**；最多缓存展示用数字，且必须清楚 **以主站为准**。

### 4.3 模型计费与用户「当前能花多少」（实时性）

- **定价 / 价目表**：可以 **批量同步、短缓存**，减少每一步都打主站。  
- **「这笔钱现在扣不扣得下」**（可用余额、是否允许开跑）要按 **实时或近实时** 来：用户刚充完、刚在别端消费完，**不能再靠过期的余额去做「能不能生成」的决定**。  
- 工程上常见做法：**发起扣费或发起任务前**，向主站做一次 **轻量校验 / 余额 introspect**（或与扣费接口合并成「预扣 + 确认」）；展示给用户的「还剩多少」可以略缓，**门禁不能与价目表同一档缓存**。

---

## 5. 管理后台财务（你现在有的 + 重构要覆盖的）

这些能力 today 分散在主站后台，重构时要 **归到「财务域」的思维**里整体梳理（界面可以还在主站，**代码与数据流要清晰**）：

| 板块 | 说明 |
|------|------|
| **定价与参考价** | 工具按次价格、方案 A 系数、学习/工具价目联动 |
| **订阅与订单** | 计划、订单、与提现审核相关 |
| **充值与优惠** | 模板、核销、活动 |
| **钱包与提现** | 用户提现申请、后台审核打款、应扣未扣 |
| **报表与核对** | 充值汇总、消耗汇总、对账用导出（和真实支付接通后的渠道流水，逐步对齐） |

**目标**：运营在后台改配置，**立刻反映在「扣款规则」上**；用户在前台与工具里 **只看到结果一致**，不出现「后台一个价、工具一个价」。

---

## 6. 核算与对账（从管理到「算得清」）

- **管理**：配置、审批、人工处理（提现、调账若将来有）。  
- **核算**：每一笔 **充值 / 消费 / 退款** 能否 **追溯到人和单**；同一笔任务 **不会扣两次**（重复提交要有防护）。  
- **对账**（随正式支付加深）：渠道实收 vs 系统入账 vs 用户余额变动 —— 这是 **财务重构后半段** 必须留口子，不必第一天全做完，但 **结构上要预留**（例如：流水号、渠道单号、幂等键）。

---

## 7. 全模态工作室（`m.ai-code8.com`）怎么接入（步骤清单）

1. **域名与证书**：与 `tool` 一样，走 HTTPS。  
2. **环境变量**（和工具站同套路）：  
   - 能回到 **主站地址**（相当于今天的 `MAIN_SITE_ORIGIN`）；  
   - **用户浏览器打开的 `m` 的网址** 要和主站里配置的「工具公用地址」规则一致（便于换票、Cookie 策略你们已在生产踩过坑，按同一套规范来）。  
3. **侧栏入口**：在工具站导航里加一项，点过去打开 **`https://m.ai-code8.com/...`**。  
4. **计费名**：在主站给工作室起好 **唯一的工具计费名**（例如 `multimodal-studio__invoke`），**先能在后台定价**，再开放扣费。  
5. **一次扣费**：任务成功后调 **主站「上报用量」类接口**（今天的做法的延续），带上 **任务 id**，避免重复扣。  
6. **充值入口**（可选）：在工作室 UI 放「余额不足去充值」，**链到主站同一套收银**；到账后主站账本更新后，**下一次门禁/扣费自然读到新余额**。  
7. **可用余额门禁**：**开跑 / 预扣前** 以主站返回为准（短 TTL 或直接请求）；**不要把「够不够」绑在和价目表一样长的缓存上**。

---

## 8. 安全（不讲行话版）

- **工具站、工作室** 都不拿 **数据库超级密码**；动账走 **接口 + 用户令牌**。  
- **令牌** 只证明「你是谁」，**不能代替**「规则怎么扣」；规则在 **财务那一侧** 统一执行。  
- **分域名** 可以 **分机器**，**不分两套账本**。  

---

## 9. 落地顺序（从稳到快）

1. **先把主站里的财务相关代码「归堆」**：钱包、扣费、价目、后台动作 —— **谁先谁后详见 checklist**，原则是先 **一条明路写账**，再拆服务。  
2. **接口整齐**：工具 / `m` 只认 **少数几个固定地址**（询价、扣费、查余额类），减少以后改一处要改十处。  
3. **再上 `m` 域名工作室**：复用换票与扣费套路，**只做新 UI 与模型管线**。  
4. **若以后业务量很大**：把「写账」那块单独起一个小进程也行，**对工具来说只是换了一个网址前缀**，业务感不变。  
5. **门禁 SLA**：产品层面区分 **价目缓存**（可几十秒级）与 **可用余额 / 是否可扣**（应到达「充值后立刻能开跑」、或明确写清可接受的秒级延迟），避免运营以为是实时的、实现却用了长缓存。

## 10. 已定 / 未定（方便你心里有数）

| 已定（工作假设） | 未定（可后续拍板） |
|------------------|-------------------|
| 新应用独立二级域，例如 `m.ai-code8.com` | `m` 与 `tool` 的 Cookie 是否需同父域等细节，实现时由前端/安全再定一版 |
| 扣款只走主站侧统一能力 | 财务小服务何时从主站进程里「拆出去」 |
| 工具站做入口与链接 | 全模态工作室首个版本包含哪些具体能力清单 |
| **`tool` / `m` 可放充值入口**，进账仍走主站 | 充值是 **深链主站整页** 还是 **嵌入式收银**，按体验与安全再选 |
| **可用余额 / 能否开跑** 按实时或近实时，**不与价目长缓存混为一谈** | 余额 introspect 与扣费是 **两请求** 还是 **合并**、允许的延迟上限（秒级） |
